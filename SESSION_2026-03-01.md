# Sesion de trabajo - 2026-03-01

Proyecto: `C:\local\microservicios\local-tts-service`  
Objetivo: montar y validar microservicio TTS local (Kokoro + Chatterbox), corregir idioma espanol, y evaluar rendimiento en RTX 5070 Ti (Blackwell `sm_120`).

## 1) Resumen ejecutivo

- Se implemento gateway OpenAI-compatible con soporte de dos motores (`kokoro`, `chatterbox`).
- Se corrigio problema de pronunciacion en espanol (propagacion de `lang_code` + voz ES en Chatterbox).
- Se ejecutaron benchmarks con metricas objetivas (`latency_ms`, `ttfa_ms`, `rtf`, `chars_per_sec`).
- Se detecto y resolvio el bloqueo critico de Blackwell:
  - Kokoro en Docker GPU fallaba por incompatibilidad `sm_120`.
  - Se habilito pipeline local no-Docker con PyTorch `cu128`, confirmando CUDA activa.
- Se midio mejora real de rendimiento de Kokoro tras mover a pipeline local Blackwell.

## 2) Cronologia paso a paso

### Fase A - Bootstrap del servicio

1. Se creo estructura del proyecto desde cero:
   - `gateway/` (FastAPI)
   - `benchmark/`
   - `scripts/`
   - `tests/`
   - `docker-compose.yml`, `.env.example`, `README.md`
2. Se implemento gateway con endpoints:
   - `POST /v1/audio/speech`
   - `GET /v1/models`
   - `GET /healthz`
3. Se anadieron adapters para Kokoro y Chatterbox.
4. Se integro chunking de texto en espanol y concatenacion de audio.

### Fase B - Correcciones tecnicas iniciales

1. Error en tests por `pydub` + Python 3.13 (`audioop` / `pyaudioop` ausente).
   - Solucion: reemplazo de flujo con `ffmpeg` (sin `pydub` en paths criticos).
2. Error de quoting en PowerShell con `curl.exe` para JSON.
   - Solucion: uso de `Invoke-WebRequest` / `Invoke-RestMethod`.
3. Validacion de servicios Docker y smoke tests de API.

### Fase C - Idioma y voces

1. Se detecto que `CHUNK_LANGUAGE=es` solo afectaba segmentacion, no fonetica.
2. Se agrego `lang_code` al request del gateway y se propago a ambos adapters.
3. Se ajustaron defaults:
   - `DEFAULT_VOICE_KOKORO=ef_dora`
   - `DEFAULT_LANG_CODE=es`
4. Se cargo voz espanola en Chatterbox:
   - voice pack `es_carlos` (desde `.ogg`)
   - `default_voice` de Chatterbox fijado a `es_carlos`
   - `DEFAULT_VOICE_CHATTERBOX=es_carlos`

### Fase D - Blackwell (RTX 5070 Ti)

1. Problema critico encontrado:
   - Kokoro Docker GPU fallaba en startup por `sm_120 not supported` y `no kernel image`.
2. Se introdujo estrategia alternativa:
   - pipeline local no-Docker para Kokoro (`venv` Python 3.12 + PyTorch `cu128`).
3. Se creo carpeta `kokoro_local/` con scripts de setup y verificacion.
4. Se verifico GPU:
   - `torch_version=2.10.0+cu128`
   - `cuda_available=True`
   - `gpu_name=NVIDIA GeForce RTX 5070 Ti`
   - `gpu_capability=sm_120`

### Fase E - Integracion operativa

1. Se anadio script para alternar backend Kokoro por `.env`:
   - `scripts/switch_kokoro_backend.ps1`
   - `local` -> `KOKORO_URL=http://host.docker.internal:8881`
   - `docker` -> `KOKORO_URL=http://kokoro:8880`
2. Se dejo `gateway` sin dependencia dura de servicio `kokoro` Docker para permitir modo local.

## 3) Errores encontrados y resolucion

### 3.1 Docker / instalacion

- Descargas muy lentas de imagenes grandes (`kokoro`, `chatterbox`), con sensacion de "bloqueo".
  - Causa: pull + verificacion + extraccion de capas grandes en Docker Desktop.
- Error Buildx en Windows:
  - `rename ...\.docker\buildx\refs\__group__... file is being used by another process`
  - Impacto: build puntual fallido/intermitente.
  - Resolucion: reintentos limpios y reconstruccion controlada de `gateway`.

### 3.2 Runtime Kokoro GPU Docker

- Error principal:
  - `NVIDIA GeForce RTX 5070 Ti ... sm_120 not compatible ...`
  - `CUDA error: no kernel image is available for execution on the device`
  - Resolucion: fallback inicial a Kokoro CPU para continuidad + migracion a pipeline local cu128.

### 3.3 PowerShell / peticiones HTTP

- `curl.exe` con JSON mal escapado:
  - `Could not resolve host`, `URL rejected`
  - Resolucion: usar `Invoke-WebRequest` / `Invoke-RestMethod`.

### 3.4 Script local Kokoro

- Colision variable PowerShell (`$Host` readonly).
  - Resolucion: renombrar parametro a `$BindHost`.
- Falta de modelo local (`kokoro-v1_0.pth` no encontrado).
  - Resolucion: ejecutar downloader de modelo en `api/src/models/v1_0`.
- Instalacion interrumpida por lock de archivo (`WinError 32` en `torch`).
  - Resolucion: matar procesos Python del mismo entorno y reinstalar.

### 3.5 Documentacion

- `README.md` quedo eliminado accidentalmente durante interrupcion de una accion.
  - Resolucion: reconstruccion completa con runbook actualizado.
